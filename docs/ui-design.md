# UI設計（P1-01 / P1-02）

この文書は、実装前タスク `P1-01: 画面構成設計（単画面）` と
`P1-02: UI挙動設計` の成果物です。
初期スコープ（`SELECT` / `FROM` / `WHERE`）に対応するための
最小UI構成と状態別挙動を定義します。

## 1. 画面目的

- SQLを1文入力し、説明生成を実行し、結果とメッセージを同一画面で確認できるようにする。
- 学習用途を想定し、操作手順を「入力 → 実行 → 結果確認」の3段階に限定する。

## 2. 対象画面

- 画面数: 1（単画面）
- 想定技術: WPF（WinFormsでも同等構成を再現可能）
- 想定ウィンドウタイトル: `SQL説明ツール`

## 3. 主要UI要素

P1-01で必須とされた4要素を以下で定義する。

1. 入力欄（SQL入力テキストエリア）
   - 複数行入力を許可する。
   - 1回の説明対象は単一SQL文とする（スコープ準拠）。

2. 実行ボタン
   - ラベル案: `説明する`
   - クリック時に入力欄の内容を解析処理へ渡す。

3. 説明表示欄
   - 生成された説明文（要約 + 句別説明）を表示する。
   - 初期表示は空、またはプレースホルダー文とする。

4. メッセージ欄
   - 空入力、未対応構文、解析不能などの状態メッセージを表示する。
   - 正常時は非表示、または空表示とする。

## 4. 画面配置（レイアウト）

視線の流れを「上から下」に統一し、学習コストを下げる。

- 上段: 入力欄
- 中段: 実行ボタン
- 下段: 説明表示欄
- 最下段: メッセージ欄

### 4-1. 簡易ワイヤーフレーム

```text
+--------------------------------------------------+
| SQL入力                                           |
| +----------------------------------------------+ |
| | SELECT id, name                              | |
| | FROM users                                   | |
| | WHERE active = 1;                            | |
| +----------------------------------------------+ |
|                                                  |
| [ 説明する ]                                     |
|                                                  |
| 説明結果                                         |
| +----------------------------------------------+ |
| | 要約: ...                                     | |
| | - SELECT: ...                                 | |
| | - FROM: ...                                   | |
| | - WHERE: ...                                  | |
| +----------------------------------------------+ |
|                                                  |
| メッセージ: （未対応構文があります）             |
+--------------------------------------------------+
```

## 5. 要素ごとの配置方針

- 入力欄
  - 画面内で最も広い領域を確保する（ユーザー入力を優先）。
  - 折り返し表示を有効にし、横スクロール依存を避ける。

- 実行ボタン
  - 入力欄直下に単独配置し、操作対象を明確化する。
  - 初期実装では追加ボタン（クリア等）を置かない。

- 説明表示欄
  - 入力欄に次ぐ広さを確保し、複数行表示を前提とする。
  - 編集不可（読み取り専用）を基本とする。

- メッセージ欄
  - 説明表示欄の下に固定し、状態通知の場所を一貫化する。
  - 1〜2行で収まる短文表示を基本とする。

## 6. 状態別UI挙動（P1-02）

P1-02で求められる4状態（実行時、空入力時、未対応時、解析不能時）を
以下のとおり定義する。

### 6-1. 実行時（正常系）

- 条件
  - 入力欄にSQL文字列があり、初期対応範囲（`SELECT` / `FROM` / `WHERE`）
    で説明可能であること。
- 挙動
  1. 実行ボタン押下で解析処理を開始する。
  2. 説明表示欄に「要約 + 句別説明」を表示する。
  3. メッセージ欄は空表示（または非表示）とする。
- 表示内容（例）
  - 説明表示欄:
    - `要約: usersテーブルから id, name を取得し、active = 1 で絞り込みます。`
    - `- SELECT: id, name を取得します。`
    - `- FROM: users を参照します。`
    - `- WHERE: active = 1 に一致する行のみ対象です。`

### 6-2. 空入力時

- 条件
  - 入力欄が空、または空白/改行のみで構成されること。
- 挙動
  1. 説明生成を行わない。
  2. 既存の説明表示欄はクリアする（前回結果を残さない）。
  3. メッセージ欄に入力促しを表示する。
- 表示内容（例）
  - メッセージ欄: `SQLを入力してから「説明する」を押してください。`

### 6-3. 未対応構文を含む時

- 条件
  - `JOIN` / サブクエリ / `GROUP BY` / `ORDER BY` など、
    初期スコープ外の構文を含むこと。
- 挙動
  1. 説明可能な範囲（`SELECT` / `FROM` / `WHERE`）は表示する。
  2. 未対応要素は詳細説明せず、注意メッセージとして通知する。
  3. メッセージ欄に「未対応構文あり」を表示する。
- 表示内容（例）
  - 説明表示欄: `注意: JOIN は初期バージョンでは詳細説明の対象外です。`
  - メッセージ欄: `未対応構文が含まれています（詳細は説明欄を確認してください）。`

### 6-4. 解析不能時

- 条件
  - SQLとして句を抽出できない、または入力が壊れていて
    初期ルールで解釈できないこと。
- 挙動
  1. 説明表示欄はクリアする（誤解を招く説明を出さない）。
  2. メッセージ欄に解析不能メッセージを表示する。
  3. ユーザーに再入力の方向性を短文で示す。
- 表示内容（例）
  - メッセージ欄:
    `SQLを解析できませんでした。SELECT / FROM / WHERE を含む単一SQL文で再入力してください。`

### 6-5. 状態遷移時の共通ルール

- 実行のたびに、説明表示欄とメッセージ欄は「今回結果」で上書きする。
- 正常結果とエラーメッセージを同時に矛盾表示しない。
  - 正常系: 説明表示あり + メッセージ欄空。
  - 異常/注意系: 説明表示（必要最小限）+ メッセージ欄に状態通知。
- メッセージは1〜2文の短文で統一し、次に取るべき操作を含める。

## 7. P1-01 / P1-02 完了判定との対応

- 主要UI要素（入力欄・実行ボタン・説明表示欄・メッセージ欄）を定義済み。
- 各要素の配置順序と目的を明記済み。
- 単画面前提で実装可能な粒度（ワイヤーフレーム含む）で記載済み。
- 主要状態（実行時、空入力時、未対応時、解析不能時）ごとの
  挙動と表示内容を定義済み。

`P1-01: 画面構成設計（単画面）` および
`P1-02: UI挙動設計` の完了条件を満たす。
