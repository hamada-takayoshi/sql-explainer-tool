# UI設計（P1-01〜P2-02）

この文書は、実装前タスク `P1-01: 画面構成設計（単画面）`、
`P1-02: UI挙動設計`、`P1-03: UI表示ルール定義`、
`P2-01: UI設計文書化`、`P2-02: UI受け入れ条件定義` の成果物です。
初期スコープ（`SELECT` / `FROM` / `WHERE`）に対応するための最小UI構成と挙動を定義します。

## 1. 画面目的

- SQLを1文入力し、説明生成を実行し、結果とメッセージを同一画面で確認できるようにする。
- 学習用途を想定し、操作手順を「入力 → 実行 → 結果確認」の3段階に限定する。

## 2. 対象画面

- 画面数: 1（単画面）
- 想定技術: WPF（WinFormsでも同等構成を再現可能）
- 想定ウィンドウタイトル: `SQL説明ツール`

## 3. 主要UI要素

1. 入力欄（SQL入力テキストエリア）
   - 複数行入力を許可する。
   - 1回の説明対象は単一SQL文とする（スコープ準拠）。

2. 実行ボタン
   - ラベル案: `説明する`
   - クリック時に入力欄の内容を解析処理へ渡す。

3. 説明表示欄
   - 生成された説明文（要約 + 句別説明）を表示する。
   - 読み取り専用とする。

4. メッセージ欄
   - 空入力、未対応構文、解析不能などの状態メッセージを表示する。
   - 正常時は空表示（または非表示）とする。

## 4. 画面配置（レイアウト）

- 上段: 入力欄
- 中段: 実行ボタン
- 下段: 説明表示欄
- 最下段: メッセージ欄

### 4-1. 簡易ワイヤーフレーム

```text
+--------------------------------------------------+
| SQL入力                                           |
| +----------------------------------------------+ |
| | SELECT id, name                              | |
| | FROM users                                   | |
| | WHERE active = 1;                            | |
| +----------------------------------------------+ |
|                                                  |
| [ 説明する ]                                     |
|                                                  |
| 説明結果                                         |
| +----------------------------------------------+ |
| | 要約: ...                                     | |
| | - SELECT: ...                                 | |
| | - FROM: ...                                   | |
| | - WHERE: ...                                  | |
| +----------------------------------------------+ |
|                                                  |
| メッセージ: （未対応構文があります）             |
+--------------------------------------------------+
```

### 4-2. 要素ごとの配置方針（粒度維持）

- 入力欄（SQL入力）
  - 画面内で最も広い領域を確保する（入力作業を最優先）。
  - 折り返し表示を有効化し、横スクロール依存を避ける。
  - 垂直スクロールを有効化し、複数行SQLでも編集しやすくする。
  - ラベル `SQL入力` は入力欄の直上に配置し、目的を明示する。

- 実行ボタン（説明する）
  - 入力欄の直下に単独配置し、操作導線を「入力→実行」で固定する。
  - 初期実装では補助ボタン（クリア/整形）は設置しない。
  - ボタン文言は動作が直感的に分かる `説明する` を採用する。

- 説明表示欄（説明結果）
  - 入力欄に次ぐ表示領域を確保し、結果確認の視認性を担保する。
  - 編集不可（読み取り専用）とし、表示専用コンポーネントとして扱う。
  - 垂直スクロールを有効化し、長い説明でも読めるようにする。
  - ラベル `説明結果` を表示欄の直上に配置し、入力欄と役割を分離する。

- メッセージ欄（状態通知）
  - 説明表示欄の直下に固定配置し、通知位置を常に一定にする。
  - 1〜2行の短文表示を基本とし、最新1件のみ表示する。
  - 正常時は空表示（または非表示）にし、異常時のみ視認できるようにする。

### 4-3. 視線導線とレイアウト意図

- 視線の流れを「上から下（入力→実行→結果→通知）」に統一する。
- 主要操作を1列レイアウトに限定し、学習用途での操作迷いを減らす。
- 状態通知を最下段に固定し、結果表示と混在しないようにする。

## 5. 状態別UI挙動（P1-02）

### 5-1. 実行時（正常系）

- 条件
  - 入力欄にSQL文字列があり、初期対応範囲（`SELECT` / `FROM` / `WHERE`）で説明可能である。
- 挙動
  1. 実行ボタン押下で解析処理を開始する。
  2. 説明表示欄に「要約 + 句別説明」を表示する。
  3. メッセージ欄は空表示（または非表示）とする。
- 表示内容（例）
  - `要約: users から id を取得し、active = 1 で絞り込みます。`
  - `- SELECT: id を取得します。`
  - `- FROM: users を参照します。`
  - `- WHERE: active = 1 に一致する行のみ対象です。`

### 5-2. 空入力時

- 条件
  - 入力欄が空文字、または空白/改行のみ。
- 挙動
  1. 説明生成処理は実行しない。
  2. 説明表示欄は更新しない（直前結果を保持）。
  3. メッセージ欄に入力促進メッセージを表示する。
- メッセージ例
  - `SQLを入力してから「説明する」を押してください。`

### 5-3. 未対応構文を含む時

- 条件
  - `JOIN` / サブクエリ / `GROUP BY` / `ORDER BY` など、初期スコープ外の構文を含む。
- 挙動
  1. 説明生成処理は中断する（部分説明は行わない）。
  2. 説明表示欄は更新しない（直前結果を保持）。
  3. メッセージ欄に対象の未対応構文を表示する。
- メッセージ例
  - `未対応の構文を検出しました: JOIN`
  - `未対応の構文を検出しました: GROUP BY`

### 5-4. 解析不能時

- 条件
  - 構文崩れなどによりSQLとして解釈できない。
- 挙動
  1. 説明生成処理は失敗として扱う。
  2. 説明表示欄は更新しない（直前結果を保持）。
  3. メッセージ欄に再入力を促すエラーメッセージを表示する。
- メッセージ例
  - `SQLを解析できませんでした。構文を確認してください。`

## 6. UI表示ルール（P1-03）

### 6-1. 初期表示

- 入力欄: 空
- 説明表示欄: 空、または `ここに説明結果が表示されます。`
- メッセージ欄: 空

### 6-2. 改行 / 箇条書きルール

説明表示欄の出力形式は以下に固定する。

1. 1行目: `要約: ...`
2. 2行目以降:
   - `- SELECT: ...`
   - `- FROM: ...`
   - `- WHERE: ...`

補足ルール:

- 1要素1行を原則とする。
- 存在しない句は出力しない。
- 不要な空行は入れない。

### 6-3. 長文入力時の扱い

- 入力欄
  - 垂直スクロールを有効化する。
  - 折り返し表示を有効化し、横スクロール依存を避ける。
- 説明表示欄
  - 読み取り専用 + 垂直スクロールを有効化する。

### 6-4. メッセージ表示ルール

- 同時に複数メッセージは表示しない（最新1件のみ）。
- 優先度は `解析不能 > 未対応構文 > 空入力 > なし` とする。
- 文言は短く、次の行動が分かる内容にする。

### 6-5. 更新ポリシー（表示保持ルール）

- 正常時: 説明表示欄を最新結果で上書きし、メッセージ欄をクリアする。
- 空入力時: 説明表示欄は保持し、メッセージ欄のみ更新する。
- 未対応構文時: 説明表示欄は保持し、未対応通知を表示する。
- 解析不能時: 説明表示欄は保持し、解析不能通知を表示する。
- 保持ポリシーの目的は「最後に成功した説明」を参照できる状態を保つこと。

## 7. 部品責務（P2-01）

UI設計文書化として、部品ごとの責務を明確化する。

- `SqlInputTextBox`（入力欄）
  - SQL文字列を受け付ける。
  - 入力はUIで保持し、処理開始時にそのまま渡す。

- `ExplainButton`（実行ボタン）
  - 説明生成イベントを発火する。
  - バリデーション結果に応じた分岐の起点となる。

- `ExplanationTextBox`（説明表示欄）
  - 説明結果（要約 + 句別説明）を表示する。
  - 編集不可で、表示専用とする。

- `MessageTextBlock`（メッセージ欄）
  - 直近の状態メッセージ1件を表示する。
  - 優先度ルールに従って表示内容を更新する。

## 8. イベント一覧（P2-01）

| イベントID | トリガー | 入力 | 処理 | 出力 |
|---|---|---|---|---|
| UI-E01 | 実行ボタン押下 | SQL文字列 | 空入力判定 | メッセージ表示または次工程へ |
| UI-E02 | `UI-E01` 後 | SQL文字列 | 未対応構文判定 | メッセージ表示または次工程へ |
| UI-E03 | `UI-E02` 後 | SQL文字列 | 解析・説明生成 | 説明表示欄更新 |
| UI-E04 | 解析失敗時 | 失敗情報 | エラーメッセージ生成 | メッセージ欄更新 |
| UI-E05 | 解析成功時 | 説明文 | 状態クリア | メッセージ欄空表示 |

## 9. エラー表示方針（P2-01）

- エラー/警告はメッセージ欄に一元表示する。
- 説明表示欄は失敗時に更新しない（誤解を避けるため直前の正しい結果を保持）。
- メッセージは以下の分類で運用する。
  - 入力エラー: 空入力
  - 警告: 未対応構文
  - 解析エラー: SQL解析不能
- 文言は固定テンプレートを基本とし、必要に応じて未対応要素名のみ差し替える。

## 10. UI受け入れ条件（P2-02）

客観判定できる最小受け入れ条件を定義する。

### AC-01: 入力→説明表示（正常）

- 前提: `SELECT id FROM users WHERE active = 1;` を入力。
- 手順: 「説明する」を押下。
- 合格条件:
  - 説明表示欄に要約が1行表示される。
  - `SELECT` / `FROM` / `WHERE` の各行が表示される。
  - メッセージ欄が空表示になる。

### AC-02: 空入力

- 前提: 入力欄を空にする。
- 手順: 「説明する」を押下。
- 合格条件:
  - 説明生成処理が実行されない。
  - メッセージ欄に入力促進文言が表示される。

### AC-03: 未対応構文

- 前提: `SELECT u.id FROM users u JOIN orders o ON u.id = o.user_id;` を入力。
- 手順: 「説明する」を押下。
- 合格条件:
  - 説明表示欄が更新されない。
  - メッセージ欄に未対応構文（`JOIN`）が含まれて表示される。

### AC-04: 解析不能

- 前提: `SELECT FROM WHERE` のような不正SQLを入力。
- 手順: 「説明する」を押下。
- 合格条件:
  - 説明表示欄が更新されない。
  - メッセージ欄に解析不能メッセージが表示される。
